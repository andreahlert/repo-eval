Hello World (README)

fail
x
README example crashes: `_LocalRun` has no `result` attribute The first Python example in the README (bundled in package METADATA) calls `run.result` but the `_LocalRun` class (defined in `flyte/_run.py`, line ~640) has no `result` property or attribute. Its available members are: `_outputs`, `url` (property), `wait()` (async no-op), and `outputs()` (returns `ActionOutputs` tuple). The correct code is `run.outputs()[0]` or `outputs = run.outputs(); print(outputs.o0)`. This is a documentation bug introduced in the 2.0 rewrite where the API surface changed but the README was not updated to match.
x
No obvious workaround without reading source code A new user hitting `AttributeError: '_LocalRun' object has no attribute 'result'` has no breadcrumb to follow. The `_LocalRun` class is private (underscore-prefixed), so `dir()` output is unhelpful. The actual method `outputs()` is decorated with `@syncify` (async-to-sync bridge), returning an `ActionOutputs` object that subclasses `tuple`. Discovering this requires reading `flyte/_run.py` source. The README should show: `result = run.outputs()[0]` or use tuple unpacking `(result,) = run.outputs()`.
!
Return type annotation is misleading The `_Runner.run()` method signature declares `-> Union[R, Run]` but `flyte.run()` always returns a `Run` object (specifically `_LocalRun` in local mode). This misleads IDE autocompletion and type checkers, compounding the discoverability problem.

Dependencies & Weight

warning
!
20 direct dependencies 20 direct deps is heavy (threshold: 15). Includes infrastructure-weight packages: grpcio, protobuf, pydantic, httpx, cloudpickle, fsspec, obstore, mashumaro, keyring. Several of these (grpcio, protobuf) are notoriously slow to compile on some platforms and increase environment setup time. For users who only need local execution, deps like grpcio and keyring add unnecessary weight.
!
`rich-click==1.8.9` exact pin risks diamond dependency conflicts The `rich-click` package is pinned to exactly `==1.8.9`. This creates hard conflicts if any other package in the user's environment requires a different version. rich-click depends on `click>=7` and `rich>=10.7`, so transitive constraints compound. Users integrating flyte into existing projects with rich or click dependencies may hit `ResolutionImpossible` errors. Should use a compatible release constraint like `~=1.8.9` instead.
!
`flyteidl2==2.0.7` exact pin blocks independent bugfix releases The protobuf IDL package `flyteidl2` is pinned to exactly `==2.0.7`, meaning any bugfix release to flyteidl2 requires a coordinated flyte release. While version-locking IDL packages is understandable for API compatibility, a `~=2.0.7` constraint would allow patch releases (2.0.8, 2.0.9) without requiring a full SDK update.
