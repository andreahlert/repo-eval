2 Hello World (README)

terminal - hello world (readme) 4.26s
x
README example crashes: `_LocalRun` has no `result` attribute
The first Python example in the package METADATA (README) calls `run.result` on line 95, but the `_LocalRun` class (defined in `flyte/_run.py:640-674`) does not expose a `result` property. The correct API is `run.outputs()` which returns an `ActionOutputs` tuple subclass. To get the scalar result: `run.outputs()[0]` or `run.outputs().o0`. This is a documentation bug shipped in the 2.0.7 METADATA, meaning every new user hitting PyPI or `pip show` will encounter broken example code on their first try.
!
ActionOutputs API is non-obvious
`flyte.run()` returns a `_LocalRun` whose `outputs()` method returns `ActionOutputs`, a `tuple` subclass. Results are accessed positionally (`outputs[0]`) or via auto-generated names (`outputs.o0`, `outputs.named_outputs['o0']`). This is not documented in the README and would require reading source code to discover after hitting the `AttributeError`.
!
No quickstart recovery path
After the README example fails, there is no `--help` hint, error suggestion, or link to docs in the traceback. A new user must read source code or search GitHub issues to find the correct API. The SDK could improve this by adding a `__getattr__` on `_LocalRun` that raises a helpful error like "Did you mean .outputs()?".

4 Dependencies & Weight

terminal - dependencies & weight 3.42s
!
20 direct dependencies
The package declares 20 direct runtime dependencies including heavyweight libraries: grpcio, protobuf, pydantic, httpx, keyring, cloudpickle, fsspec, and obstore. This is heavy for a first install and increases the surface area for version conflicts in existing projects.
!
`flyteidl2==2.0.7` exact pin risks conflicts
`flyteidl2` is pinned to exactly `2.0.7` (lock-step with the SDK version). This means any environment that needs a different flyteidl2 version (e.g., for another Flyte-ecosystem tool) will hit an immediate resolver conflict. Lock-step pins are common for protobuf IDL packages but create friction in monorepos or shared virtualenvs.
!
`rich-click==1.8.9` exact pin risks conflicts
`rich-click` is pinned to exactly `1.8.9`. Any other package in the environment requiring a different rich-click version will cause pip to fail. This pin is likely due to API compatibility concerns but should ideally be a compatible-release constraint (`~=1.8.9`) instead of an exact pin.
+
54 total packages in venv
Transitive dependency count is 54, well under the 80 threshold.
