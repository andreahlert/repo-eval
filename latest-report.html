Hello World (README)

fail
x
README example crashes: `_LocalRun` has no `result` attribute The first Python example on the README calls `run.result` but the `_LocalRun` class (defined as an inner class inside `_Runner._run_local` in `flyte/_run.py:640-674`) does not expose a `result` property. The return value is stored in `self._outputs` and only accessible via the `outputs()` method (an async method wrapped with `@syncify`). The correct code should be `run.outputs()` instead of `run.result`. This is a documentation bug: the API changed but the README was not updated to match. This is the single worst adoption barrier since it breaks the very first thing a new user tries.
!
No obvious workaround in error message The `AttributeError: '_LocalRun' object has no attribute 'result'` gives no hint about what the correct attribute is. A new user would have to read source code or guess that `outputs()` is the replacement. The `Run` base class and `_LocalRun` subclass could benefit from a `__getattr__` hint or a `result` property alias pointing users to `outputs()`.
!
README example uses async task without async runner context The example defines `async def main(...)` as a task and calls it via `flyte.run()` which is itself `@syncify`-wrapped. While this works mechanically, it obscures the sync/async boundary for newcomers. The example mixes `asyncio.gather`, `@syncify`, and bare `flyte.run()` without explaining the execution model.

The `flyteidl2` pin is understandable (protobuf IDL must match SDK version), but `rich-click==1.8.9` is a UI library with no API stability reason for an exact pin. This can cause `ResolutionImpossible` errors when another package in the user's environment needs a different rich-click version.
